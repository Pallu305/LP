# 🔷 Practical 7

## **Scale a Kubernetes deployment and verify the status of pods and services**

---

# 🔹 Aim

To scale a Kubernetes deployment and verify the status of pods and services.

---

# 🔹 Theory

### 🔸 Scaling

* Scaling means **increasing or decreasing number of pods**
* Helps handle more user traffic

---

### 🔸 Replica

* Number of pod copies running in cluster
* Controlled using deployment

---

### 🔸 Command

```bash
kubectl scale deployment <name> --replicas=<number>
```

---

# 🔹 Requirements

* Kubernetes enabled
* kubectl installed

---

# 🔹 Execution Steps

## 🔷 Step 1: Create Folder

```bash
mkdir k8s-scale-demo
cd k8s-scale-demo
```

---

## 🔷 Step 2: Create Deployment

```bash
kubectl create deployment scale-app --image=nginx
```

### ✅ Expected Output:

```bash
deployment.apps/scale-app created
```

---

## 🔷 Step 3: Check Deployment

```bash
kubectl get deployments
```

### ✅ Expected Output:

```bash
NAME        READY   UP-TO-DATE   AVAILABLE
scale-app   1/1     1            1
```

---

## 🔷 Step 4: Check Pods

```bash
kubectl get pods
```

### ✅ Expected Output:

```bash
NAME                        READY   STATUS
scale-app-xxxxx-xxxxx       1/1     Running
```

---

## 🔷 Step 5: Scale Deployment (Increase Pods)

```bash
kubectl scale deployment scale-app --replicas=3
```

### ✅ Expected Output:

```bash
deployment.apps/scale-app scaled
```

---

## 🔷 Step 6: Verify Pods After Scaling

```bash
kubectl get pods
```

### ✅ Expected Output:

```bash
NAME                        READY   STATUS
scale-app-xxxxx-xxxxx       1/1     Running
scale-app-xxxxx-yyyyy       1/1     Running
scale-app-xxxxx-zzzzz       1/1     Running
```

👉 Now **3 pods running**

---

## 🔷 Step 7: Verify Deployment

```bash
kubectl get deployments
```

### ✅ Expected Output:

```bash
NAME        READY   UP-TO-DATE   AVAILABLE
scale-app   3/3     3            3
```

---

## 🔷 Step 8: Expose Service

```bash
kubectl expose deployment scale-app --type=NodePort --port=80
```

### ✅ Expected Output:

```bash
service/scale-app exposed
```

---

## 🔷 Step 9: Verify Service

```bash
kubectl get services
```

### ✅ Expected Output:

```bash
NAME        TYPE       CLUSTER-IP     PORT(S)
scale-app   NodePort   10.x.x.x       80:3xxxx/TCP
```

---

# 🔹 Output

* Deployment created successfully
* Pods scaled from 1 → 3
* All pods in **Running state**
* Service created successfully

---

# 🔹 Working Explanation

1. Deployment initially creates 1 pod
2. Scaling command increases replicas to 3
3. Kubernetes automatically creates new pods
4. Service exposes all pods
5. Load is distributed across pods

---

# 🔹 Conclusion

Kubernetes deployment was successfully scaled, and the status of pods and services was verified.

# 🔷 Practical 7 — Step Explanation

## **Scale a Kubernetes deployment**

---

## 🔹 Step 1: Create Folder

```bash
mkdir k8s-scale-demo
cd k8s-scale-demo
```

### ✔ Why?

* Keeps your work **organized**
* Not required by Kubernetes, but good practice for labs

---

## 🔹 Step 2: Create Deployment

```bash
kubectl create deployment scale-app --image=nginx
```

### ✔ What happens?

* Kubernetes creates a **deployment named `scale-app`**
* It automatically:

  * Pulls **nginx image**
  * Creates **1 pod**
  * Starts container inside pod

👉 Think: *Deployment = manager of pods*

---

## 🔹 Step 3: Check Deployment

```bash
kubectl get deployments
```

### ✔ Why?

* To confirm deployment is created
* Shows:

  * READY pods
  * AVAILABLE pods

👉 Example:

```
1/1 → 1 pod running
```

---

## 🔹 Step 4: Check Pods

```bash
kubectl get pods
```

### ✔ Why?

* To see actual **running containers**

👉 Output shows:

* Pod name
* Status (Running / Pending)

👉 Must be:

```
STATUS = Running
```

---

## 🔹 Step 5: Scale Deployment

```bash
kubectl scale deployment scale-app --replicas=3
```

### ✔ What happens?

* You tell Kubernetes:
  👉 “I want 3 pods instead of 1”

* Kubernetes:

  * Creates **2 new pods automatically**

👉 No manual creation needed

---

## 🔹 Step 6: Verify Pods

```bash
kubectl get pods
```

### ✔ Why?

* To confirm scaling worked

👉 Now you should see:

```
3 pods running
```

👉 This is the **main proof of scaling**

---

## 🔹 Step 7: Check Deployment Again

```bash
kubectl get deployments
```

### ✔ Why?

* Confirms desired state achieved

👉 Output:

```
3/3 → 3 pods ready
```

---

## 🔹 Step 8: Expose Service

```bash
kubectl expose deployment scale-app --type=NodePort --port=80
```

### ✔ What happens?

* Creates a **service**
* Connects all pods to one access point

👉 Without this:
❌ You cannot access app externally

---

## 🔹 Step 9: Verify Service

```bash
kubectl get services
```

### ✔ Why?

* To confirm service exists

👉 Shows:

* Service name
* Port mapping

---

# 🔹 Overall Flow (Understand This)

👉 Kubernetes does:

1. Deployment created
2. Pod created
3. You scale → more pods created
4. Service connects all pods
5. Traffic distributed automatically

---

# 🔹 Real-Life Analogy

* Deployment = Manager
* Pods = Workers
* Scaling = Hiring more workers
* Service = Reception desk (entry point)

---

# 🔹 Final Understanding (Important)

👉 Scaling means:

* Not modifying pod
* Not restarting container

✔ It means:
➡️ **Increasing number of identical pods**
